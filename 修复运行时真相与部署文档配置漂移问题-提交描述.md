## 提交背景

本次修改针对审计报告中已经证实的一类系统性问题：

> **运行时真相与文档 / 配置 / 部署模型严重漂移**

在修改前，仓库存在以下典型分裂：

1. `README.md`、`.env.example`、`config/bot.config.mjs`、`Dockerfile`、`docker-compose.yml` 描述的是一套 **env 驱动、`/data` 落盘、callback 可配置、reply 目录明确** 的部署模型。
2. 但真实入口 `src/qq_bot.mjs` 仍由大量 `EXAMPLE_*` 硬编码主导，包括：
   - owner / bot QQ
   - monitored groups
   - Gateway 地址 / token
   - callback 端口与绑定地址
   - 数据目录与 reply 目录
   - systemd runtime user
   - benchmark 的日志路径与 gateway 端口假设
3. `scripts/setup_cron.sh` 与主程序对 `.env` 的读取语义也不一致，导致同一份配置文件在在线与离线链路中可能被不同方式解释。

这类问题最危险的地方不在于单个变量填错，而在于：

- 用户按 README 部署，程序却不一定按 README 运行；
- Docker / Compose 看起来支持一套路径和网络模型，但主程序实际走的是另一套；
- `.env` 中看似能配置的项，在线链路可能完全不生效；
- 排障时如果按文档去找 callback / reply / data 目录，极易被错误信息误导。

本次提交的目标，就是把这些“看起来像配置，实际却不生效”的裂缝收起来，让 **代码、文档、部署、离线脚本** 回到同一个 source of truth。

## 本次提交包含的主要改动

### 1. 让 `src/qq_bot.mjs` 真正接入 `config/.env`

将主程序中原本散落的 `EXAMPLE_*` 与固定常量替换为从 `config/bot.config.mjs` 导入的运行配置，接线范围包括：

- `BOT_QQ`
- `OWNER_QQ` / `OWNER_QQS`
- `OWNER_NAME` / `BOT_NAME` / `BOT_PERSONA`
- `NAPCAT_WS_URL`
- `GATEWAY_HOST` / `GATEWAY_PORT` / `GATEWAY_TOKEN`
- `CALLBACK_HOST` / `CALLBACK_PORT`
- `DATA_DIR` 派生的上下文、日志、reply 目录
- `RUNTIME_USER`
- `OPENCLAW_LOG_DIR`
- `INTENT_PRESETS` / `QUICK_REPLY_PRESETS` / `MODEL_PRESETS`

这一步让运行时真相第一次真正回到了配置层，而不是继续由主文件里的一套示例值统治。

### 2. 统一路径模型到 `/data` + `workspace*/qq_replies`

修正了配置、主程序、容器文件与文档对数据目录和 reply 目录的描述，使它们一致采用：

- `DATA_DIR=/data`
- `WORKSPACE_DIR=/data/workspace`
- 主 reply 目录：`workspace/qq_replies`
- 其他 agent workspace reply 目录：
  - `workspace-lite/qq_replies`
  - `workspace-strong/qq_replies`
  - `workspace-heavy/qq_replies`

同时废弃掉 README / config 中原先那套误导性的 `delivery-queue` 叙事，只保留 README 中一句迁移提示，帮助使用者从旧认知过渡。

### 3. 修正 callback 暴露模型

之前 Dockerfile / Compose 会让人误以为 callback 端口应该对外暴露，但代码实际上默认只监听 loopback。现在做了两件事：

1. 主程序改为显式使用 `CALLBACK_HOST` / `CALLBACK_PORT`
2. Docker / Compose / README 全部改成：
   - callback 默认仅监听本机
   - Compose 不再额外暴露 `19283`
   - Dockerfile 不再 `EXPOSE 19283`

这样部署叙事就不再诱导使用者把一个本地辅助接口当成公网回调端口。

### 4. 收拢 owner / runtime user / benchmark 相关硬编码

除基础路径与连接配置外，还清理了几类容易被忽略但会直接影响可运维性的示例值：

- owner 识别从单个示例 QQ 切为 `OWNER_QQS`
- `systemctl --user` 操作改为走 `RUNTIME_USER`
- benchmark / model switch 不再写死 `example`
- benchmark log 路径改为使用 `OPENCLAW_LOG_DIR`
- benchmark 的 gateway 端口释放与 TCP 探测逻辑改为使用 `GATEWAY_HOST` / `GATEWAY_PORT`

这一步的价值不只是“更优雅”，而是避免线上环境表面可配置、实际仍有隐藏固定值把行为拉回示例机。

### 5. 修复 cron 与主程序对 `.env` 的解析漂移

这是这次修改里一个很关键但容易被低估的点。

原本：

- 主程序通过 `dotenv` 读取 `.env`
- `setup_cron.sh` 通过 `grep | xargs` 暴力导出 `.env`

这会导致：同一个 `.env` 文件在在线与离线链路里，可能被两套不同语义解释。

本次新增 `scripts/run_with_env.py`，让 cron 任务改为：

- 由 `run_with_env.py` 读取项目根 `.env`
- 按更接近 dotenv 的方式解析 `KEY=VALUE`
- 再带着这些变量执行 `feedback_monitor.py` / `learning_summarizer.py`

同时 `setup_cron.sh` 也从“grep 判重”升级为 marker block 方式，保证：

- 重复运行不会重复安装
- 原有受控块会被整块替换
- 用户自有 cron 不会被无差别污染

### 6. 修正文档中仍残留的旧模型描述

README 不只是补充了新配置项，还把几处“看似小，但会误导维护者”的残余漂移一起收口：

- 命令权限说明不再一刀切写成“仅 Owner 可用”
- 命令表改为与当前代码分支一致
- 意图分级从旧的 `0-5` / `4-5` 改为真实的 `REJECT / 0 / 1 / 2 / 3 / 4`
- `AGENT_PROFILES` 字段说明改成真实支持的结构：
  - `label`
  - `tier`
  - `agentId`
  - `maxAgentEvents`
- 明确 `persona/*.example` 是模板，不是唯一运行时真相

这部分的核心目标是：**README 不再继续撒谎。**

## 提交价值总结

本次提交的核心价值不只是“改了几份配置”，而是完成了一次 **source of truth 收拢**：

1. 让主程序真正服从配置，而不是继续被示例硬编码绑架。
2. 让 Docker / Compose / README 对部署模型说同一种话。
3. 让在线主进程与 cron 离线任务对 `.env` 的理解尽量一致。
4. 让 reply 目录、callback 暴露方式、owner 模型、runtime user、benchmark 行为这些高频排障点都回到可解释、可配置、可维护的状态。

对于这个仓库来说，这次提交修复的不是某一个 bug，而是：

> **“代码在跑一套，文档在讲一套，部署文件又暗示第三套”**

这种系统性维护风险。
